3.1 User administration

In the Daimler R&D, Supplier and After-Sales environments multiple users can share the same diagnostic testing machine and as such the same ZenZefi instance. Therefore ZenZefi supports a registration and login process for multiple users, so that the users can request their individual certificates and import them in their user-specific certificate stores. All actions the user triggers via REST API, CLI or UI are related to the certificates and the configuration of the currently logged in user only.

The following user types exist:

  • Default user

The Default User can utilize ZenZefi the same as the registered user without logging into ZenZefi. However, the default user cannot retrieve certificates from the PKI.

  • Registered user

Registered users must first login into ZenZefi in order to work with their personalized ZenZefi profile and their certificates. The users who want to retrieve certificates from the Daimler Backend must officially be registered at the Daimler PKI.

3.2 User login & logout

3.2.1 Login for Daimler employees and Suppliers

1.  Press "Login" -> "Online login". The login dialog for Daimler Employees will be displayed.

2.  For Daimler employees: the user enters user ID and password (Who is Who credentials).

For Suppliers: the user enters user ID and password (Daimler Supplier credentials). If the user ID and the password match, the user is authenticated and a token will be supplied. If the user ID or the password or the combination of both do not match the user will not be authenticated and no token will be supplied.

If the user ID and the password match, the user is authenticated. If the user ID or the password or the combination of both do not match the user will not be authenticated.

  

  Figure 16: ZenZefi Backend-Login For Daimler Employees

 

3.  If a user has successfully logged in, ZenZefi checks if an user certificate store exists for the logged in user. ZenZefi behaves as follows:

  • A user certificate store does not exist:

    • A Local-Password is initially requested. This is done in a dialog where the user enters the Local-Password twice. If both Local-Passwords match and comply with the Daimler password policy, see Daimler Password Policy, the user certificate store will be created and encrypted with the Local-Password.

    • If the Local-Passwords do not match, an error "Error: Password does do not match the repeated password" will be displayed.

    • The Local-Password request dialog remains open until both passwords are identical and comply with the Daimler password policy.

    • If the "Cancel" button is pressed the user will not be logged in and the access token shall be discarded. The Certificates page of the default user will be displayed.

    • If the user certificate store has been created, the Root- and Backend CA certificates will be copied into it

  

  Figure 17: ZenZefi Local Password For Daimler Employee

 

i

Daimler Password Policy

  • Length of password: minimum 10 characters

  • Use at least one numerical digit

  • Use at least one special characters

  • Use at least one lower-case letter

  • Use at least one upper-case letter

i

Prevent brute force password attacks to ZenZefi local user certificate store

  • A user certificate store already exists:

    • If the "Cancel" button has been pressed, the user will be logged out and the access token is discarded. The default page will be displayed.

3.3.6 ECU certificate folder

The ECU certificates are grouped per backend in a separate tree node (henceforth termed also "folder") inside the certificates table, so that the visibility and handling of the ECU certificates in user certificates store is improved.

Whenever an ECU certificate is imported into ZenZefi or provided by the PKI, ZenZefi checks if an ECU folder already exists for that particular backend to which the ECU authority key identifier points to. If no ECU folder exists, ZenZefi creates one for the ECU backend. The imported ECU certificate is grouped and displayed under that ECU folder. Each ECU folder can contain maximal one ECU folder at each moment in time. All ECU certificates stored in the ECU folder are sorted and ordered by “Subject”. An ECU folder contains ECU certificates (with and without private keys) only.

Details view for an ECU folder contains the following information: ECU certificate folder for Authority Key Identifier, number of ECU certificates with private key, number of ECU certificates without private key and total number of ECU certificates.

  Figure 47: Certificates - ECU Certificate Folder

 

If an ECU folder is deleted, all ECU certificates that are grouped and displayed under that tree node will also be deleted for that particular ECU folder. If the last ECU certificate inside that ECU folder is deleted, the ECU folder will be deleted automatically for the affected backend, and should not be displayed under the backend, as long as, no further ECU certificates are imported.

i

3.5.1 Validate certificate

  Figure 50: Settings - Validate Certificate Switch

Whenever a certificate is imported or exported via REST API, CLI or UI, ZenZefi performs a Basic validation by default and an additional "Extended validation" (if set in user specific configuration).

When "Validate certificate" is OFF, Zenzefi performs Basic validation:

  • whether the imported file can be parsed to a ASN.1 structure

  • whether certificates are of a known type, which is the case when the certificate type is one of the types mentioned above, in section 3.3 "Certificates" tab

  • PKI-role extension exists and contains one of the values defined in MSS10824 chapter 5.2

  • if DELETE_EXPIRED_CERTS=true, expired certificates will not be imported. If DELETE_EXPIRED_CERTS=false, expired certificates will be imported

  • user specific configuration parameter VALIDATE_CERTS defines whether extended validation (VALIDATE_CERTS=true) has to be executed or validation is to be terminated after basic validation

When "Validate certificate" is ON:

  • whether certificate is valid with regard to MSS10824 chapter 8 specification, that means ZenZefi checks for the mandatory fields in the certificates (see in MSS10824 chapter 8, for each certificate type differently)

  • it is checked that field types are correct in the certificates (see MSS10824), e.g. that the correct length and data types are contained in the fields

  • check if certificate is expired

  • check if certificate (except enhanced rights and SecOcIS certificates) contains a public key

  • check chain of trust (can exist in user certificate store and import list)

1.  Root CA Certificates are checked by verifying the signature with the public key of the already available Root CA Certificate

2.  Link Certificates are checked by verifying the signature with the public key of the cross-certified Root CA and Backend CA Certificate

3.  ZenZefi checks the chain of trust of other certificates by checking the signature of the certificate to be validated with the public key of the confirming certificate

  • for all certificates except root, backend, link, ECU and enhanced rights certificates check if the private key exists in the user certificate store or in the import package (if import package is a PKCS#12 container)

  • check if for an enhanced rights certificate the corresponding diagnostic authentication certificate exists in user certificate store or in import list

 

3.5.2 Delete certificates automatically

  Figure 51: Settings - Validate Certificate Switch

3.5.3 Automatic certificate selection

  Figure 52: Settings - Automatic Certificate Selection

If more certificates in the certificate store match the search criteria in the REST API function calls,  ZenZefi decides which certificates shall be sent to application, according to following rules:

  • when "Automatic certificate selection" is ON:

    • All invalid certificates (expired) are ignored when searching

    • Empty/no present Target ECU in request matches empty Target ECU in certificate

    • Empty/no present Target VIN in request matches empty Target VIN in the certificate

    • Not empty Target ECU in request matches in certificate empty Target ECU or the exact value specified in request

    • Not empty Target VIN in request matches in certificate empty Target VIN or the exact value specified request

    • Empty Target ECU has higher priority than empty Target VIN. Example: if certificate 1 has an empty Target ECU and a valid Target VIN and certificate 2 has a valid Target ECU and an empty Target VIN, certificate 1 will be found in the search result

    • User role limits the searching to those certificate which have the specific user role

    • Empty Target ECU or VIN has higher priority than the validity ( In algorithm flow validity is after filter by target ECU and VIN)

    • Signature alphabetical order has lower priority then the validity

3.5.5 Priority

This option is available only if Automatic certificate selection is chosen!

There are different user priorities for the certificate selection depending on the different user roles.

Display of "User Role Priority" in Settings tab:

  • If automatic certificate selection (“Automatic certificate selection” is ON) has been chosen in Settings, the table "Priority" fades in. The user can modify the "Priority" Settings.

  • If manual certificate selection (“Automatic certificate selection” is OFF) has been chosen the table "Priority" fades out. The user cannot modify the user priority Settings. All other options are shifted up.

  • The described feature applies for both the Default user and the Registered user.

  • The "Priority" settings remain unchanged, when the user toggles between manual and automatic certificate selection. If the search result contains two or more certificates which matches the search criteria, then the one with the highest priority from this priority list is returned.

  3.5.6 Proxy

  

  Figure 56: Settings - Proxy Type

The users who are using ZenZefi within a corporate network need, in most cases, to adapt their proxy. The proxy setting is individual to each network. Users must request the proxy settings from their IT administrators. If the proxy is not set correctly, no user certificates can be retrieved from the PKI and an error "401 Not authorized. Please see log files for more details" will be returned upon login.

The user can chose between 3 types of proxy:

  • Proxy type - Manual - ZenZefi will use only dedicated proxy settings.

  • Proxy type - System (default) - ZenZefi will use only settings provided by the OS.

  • Proxy type - Direct - ZenZefi will use all the connections without having to go through a proxy server.

     Figure 57: Settings - Proxy Type Manual

 

!

The proxy settings explained above, affect only ZenZefi server not the browser settings.

The proxy settings for browser and ZenZefi server should be in sync (e.g. by using Proxy type - Native in the ZenZefi proxy settings).

 

The user can specify the proxy settings in Xentry's proxy configuration. Therefore, the proxy settings feature is disabled only from ZenZefi After-Sales.

 

3.5.7 User accounts

The user can delete any user account from the "User accounts" table, including his own and other user accounts. If a user deletes his own or other user account, a security message is displayed. Once a user account has been deleted, it is not possible to recover the user account again. Deleting a user account might be necessary, if a user has either lost his local password or the user wants to remove all certificates from the PC due to security reasons.
